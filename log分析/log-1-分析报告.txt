log-1.txt 分析报告
==================

背景：log-1.txt 是一次R70发卡弯实测记录（BT print on + track on），
对应 track_module.cpp 的调试log（T时间戳 + 5路W/B图案 + 判定模式 + 左右PWM）。


1. 核心结论（更正）：日志末尾丢线是场地没线了，不是转弯失败
--------------------------------------------------------------

【更正说明】：最初分析把log末尾的丢线判断为"R70急弯冲出赛道失败"，
用户确认实际情况是——现场赛道因场地有限，铺设的线在这里就结束了，
后面本来就没有线，所以丢线、进而LOST_STOP停车是正常现象，不是bug、
也不代表转弯没过去。

重新看时间线：T 112176~112326 是一段 SHARP_L（WBBBB），车执行了一次
急转动作；紧接着 T 112356 开始进入 LOST_L，之后5路传感器持续全黑
（BBBBB）直到log结束。结合"赛道到此为止"的信息，更合理的解读是：车
先完成了这次SHARP_L急转（大概率就是R70发卡弯本身），随后开出了赛道
铺线的物理范围，自然再也读不到白线，1500ms（LOST_TIMEOUT_MS）超时后
按设计触发LOST_STOP停车——这段行为反而说明超时保护逻辑工作正常。

结论：不能仅凭这个log断定R70急弯没过去；从"急转动作已发生、后面无线
可寻"的现象看，这次实测更可能是转弯成功、只是场地线不够长把它记录成
了丢线。待完成清单第2项（R70能否过弯）需要用一条线更长/收尾处理好的
赛道重新测一次才能下结论。


2. CH6阈值仍建议重新校准，但不再归因为"急弯失败的根因"
--------------------------------------------------------

sensor_module.cpp 里阈值 THRESHOLD = {1545, 1580, 1422, 1400, 1115}，
查git历史，这组值从最初引入到现在一直没变过，包括commit 7b24220（传感器
挪位置+180°反装）之后也没改，对应"待完成"清单第1项（传感器离地高度变了
需要重新校准）确实还没做，这一点本身仍然成立、建议继续按计划做。

看log里的原始ADC数据，CH6（index4）：
- 直道黑色基线：约2020~2550
- 之前几次成功触发SHARP的白色读数：约600~1000
- 中点应该在~1400~1600左右（跟之前记录的CH6校准值~1500吻合）

代码里CH6阈值是1115，比这个中点明显偏低（偏向"黑"）。但既然这次日志
末尾的丢线已确认是场地没线了，就不能再把这个阈值偏低当成"导致这次冲出
赛道"的证据——两者目前看是两件独立的事：阈值确实该重新校准，但没有证据
表明它在这次实测里造成了转弯失败。

log里能看到一处具体的原始数据现象（与转弯成败无关，只是记录）：
  [560] 2546B 2285B 2189B 2283B 1427W  pos:0.00
CH6原始值1427，按当前阈值1115应该判"B"（黑），但打印显示的是"W"
（原因见下面第3点，是打印代码采样时序问题，不是控制逻辑问题）。


3. 顺带发现一个打印代码的小bug（不影响巡线控制，但会误导人工调阈值）
--------------------------------------------------------------------

sensor_debug.ino 打印循环（print on时每200ms一次）里，独立调用了3次对
5路ADC的整体重新采样：

  sensor_read(vals);              // 第1次采样 -> 用于打印的原始值
  sensor_binary(is_white);        // 第2次采样 -> 用于打印的W/B标记
  float pos = sensor_position();  // 第3次采样 -> 用于打印的pos

三次采样是三个不同时刻的独立ADC读数，中间有电机PWM噪声，理论上会不一致。
上面提到的"1427却标W，同一行pos却是0.00（第3次采样时其实全黑丢线）"
就是这个bug的实锤——vals和标记W/B对不上号，pos又是第三份数据。

这只影响人工看log校准阈值时的准确性，track_update()本身每次只调用一次
sensor_binary，控制逻辑是自洽的，没受影响。


4. 建议顺序
-----------

1) 先按清单第1项，重新对CH2~CH6全部实测校准阈值（尤其CH6，当前1115
   大概率偏低），threshold CH VALUE + save——这个独立于本次log的丢线
   结论，本来就该做。
2) 顺手把sensor_debug.ino那段打印改成只采样一次、复用同一份vals[]算
   is_white[]和pos，避免log自相矛盾。
3) 想真正确认R70能不能过弯，需要换一段线足够长（转弯之后还留有一截
   直线）的赛道重测一次，这样丢线才能明确归因于"转弯失败"而不是"线
   到头了"。
